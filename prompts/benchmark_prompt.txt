Explicitly disallowed alternatives:

- Do not use Tailwind CDN, PostCSS-only setup, django-tailwind, Bootstrap, or hand-written CSS as a replacement for the official Tailwind CLI. Include source CSS, built static CSS, and the exact CLI build command.
- Do not import or use the direct `ollama` Python package for the app's LLM path. Use only `langchain_ollama.ChatOllama`; tests may fake the project-owned service boundary.
- Do not write secret-shaped placeholder values such as dummy `SECRET_KEY`, `DJANGO_SECRET_KEY`, tokens, or passwords in source, README, `.env*`, Dockerfile, or compose. Show environment variable names and commands that generate values instead.

Use the tools available in this runtime. If a tool call fails, recover and continue implementing. Do not depend on a specific agent harness tool name.

Use Django's `startproject` and `startapp` generators to scaffold the project. Configure only the apps needed for the chat UI plus Django Channels. The chat history can live in memory per browser session or per consumer; a database is not required for the chat itself, but Django's default sqlite database is fine if migrations are needed for framework boot.

Use normal Django project structure. Avoid putting all logic into one template, one consumer, or one static file. Split the LLM client into a small service module that the WebSocket consumer calls. Use Django template partials/includes for repeated UI fragments. Keep production code free of test fakes.

Architecture requirements (maps to audit D7):
- Expose the LLM service through a dedicated module with a `typing.Protocol` (or ABC) interface; the consumer depends on the protocol, not ad-hoc functions.
- When Docker/production settings exist, split settings into at least `base` + `production`/`docker` modules (or equivalent) instead of one monolithic `settings.py`.
- Keep the WebSocket consumer under ~30 non-blank lines by delegating streaming to the service layer.

Security and config requirements:
- Django `SECRET_KEY` must come from `DJANGO_SECRET_KEY` or `SECRET_KEY` with no hardcoded fallback and no placeholder literal in source, Dockerfile, compose, or docs.
- Docker/production `DEBUG` must default to `False`; local development may enable it only through an environment variable.
- `.env.example` must **not** set `DEBUG=True` or `DJANGO_DEBUG=True`; use `False`, omit the variable, or document that debug is enabled only via a local untracked `.env`.
