Prompt-Version: meta-v1.0

You are running a META-ANALYSIS over a set of code-audit reports that already exist on disk. You are NOT auditing the underlying generated projects yourself — that work is already done. Your job is to compare the audits and surface harness- vs model-attributable signal.

## Inputs (read these with the Read / Glob tools)

- Audit-reports root: {audit_reports_dir}
- Comparison table (raw, regex-parsed): {comparison_table_path}
- Statistical summary (auto-computed harness/model/dimension rollups): {statistical_summary_path}
- Per-target reports (one per audited project):
  - Glob: {audit_reports_dir}/*/*/report.md
  - Each path is shaped `<auditor>/<harness>-<model_slug>/report.md`
  - Inside, the rubric layout is sections A-I (see prompts/audit_prompt_template.txt for the schema)

The harness prefixes are: `claude`, `codex`, `opencode`. Strip the prefix to get the underlying target model slug. Targets with the same model slug under different harness prefixes are paired comparisons (same model, different agent harness).

## What you are testing

The PRIMARY question is: **which agent harness produces better Phase-1 output on the same brief?** Secondary question: are there model-attributable patterns that show up regardless of harness?

You are NOT testing the auditor model. Treat the audit scores as ground truth for this analysis; do not re-grade.

## Workflow

1. Read the comparison table and statistical summary first — they are pre-aggregated context.
2. Glob all `report.md` files. Read every one. Skim quickly when the report is short; read closely when it diverges from the cohort average.
3. Cross-tabulate by harness (rows) × dimension (columns) using the per-report scores; this is your evidence base, NOT a single global average.
4. For every model slug that appears under multiple harnesses, build a side-by-side comparison and identify which dimensions moved.
5. Identify dimensions where every harness scores poorly — those are universal blind spots (prompt-level or model-family-level issues, not harness issues).
6. Identify dimensions where one harness uniquely fails — those are harness-attributable.

## Required output

Write a single markdown file to: **{output_path}**

Use the Write tool with that exact absolute path. The file MUST contain these sections, in order:

### 1. Executive summary (2-4 sentences)

Which harness wins on average and by how much. Single most important harness-attributable pattern. Single most important universal blind spot.

### 2. Harness ranking

| Rank | Harness | Avg Total | n Runs | Tier distribution | Headline |
|---|---|---:|---:|---|---|

The headline column is a one-sentence summary of that harness's behavior across the cohort, citing one specific report:line as evidence.

### 3. Cross-harness model pairings

For every model that appears under >=2 harnesses, one subsection:

#### `<model_slug>`

- harness A: total/100, dimensions that scored lower than the model's median under other harnesses
- harness B: total/100, dimensions that scored higher than the model's median under other harnesses
- 1-sentence explanation of where the harness moved the needle, citing report:line

If a model only appears under one harness, do NOT include it here.

### 4. Dimension-level signal

For each of the 8 rubric dimensions (D1-D8), one line:

- **D<n> — <label>** (max <max>): universal-avg X.X; per-harness avg claude=Y.Y / codex=Z.Z / opencode=W.W. Classify as **universal blind spot** (all harnesses within 1 point of each other AND below 70% of max) or **harness-attributable** (any harness deviates by >=2 points from another) or **clean** (everyone scored at/near max).

### 5. Headline findings (3-5 bullets)

Each bullet:
- Cites specific report paths (e.g. `audit-reports/<auditor>/codex-glm_5_1_ollama_cloud/report.md` line N)
- States the pattern in one sentence
- Names the affected harnesses/models

### 6. Recommendations

3-5 concrete, actionable recommendations for the benchmark owner. Each one says either "to improve the harness" or "to improve the prompt" or "to retry the model under X harness". No vague advice.

### 7. Appendix: data inputs

List the comparison table path, statistical summary path, and the number of per-target reports you actually read. If any report.md was empty or malformed (e.g. the auditor stalled before writing it), name it here.
